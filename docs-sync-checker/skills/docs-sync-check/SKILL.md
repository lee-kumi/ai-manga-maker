# SKILL: docs同期チェック

## 役割
アプリ本体（HTML）と対の `docs/` 文書が、CLAUDE.mdの「二重管理」ルール通りに
そろっているかを点検し、ズレ疑いを人間に報告する。

## 動くきっかけ
- 作業ブランチで push / プルリクエストを作る前
- 「同期チェックして」「ズレ見て」と頼まれたとき
- cover-maker.html・v2.html・docs/・スキル・CLAUDE.md を触った直後

## 手順（動詞で3〜7歩）
1. **読む**：`knowledge/sync-pairs.md`（ペア表）と CLAUDE.md を読む。
2. **調べる**：`git diff --name-only main...HEAD`（＋未コミット `git status`）で、今回変わったファイルを一覧する。
3. **照合する**：ペア表の各ペアについて「片方だけ変わって相手が未変更」を探す。
4. **判定する**：各ペアを OK / 要確認 に振り分け、要確認には該当ファイル名と理由を添える。
5. **書き出す**：結果を `output/日付_sync-check.md` に保存する。
6. **報告する**：初心者向けに、要確認の数と「次にやること」を短く伝える。

## 参照するナレッジ
- `knowledge/sync-pairs.md`（二重管理ペア表・正本の定義）
- `../../CLAUDE.md`（会社の地図。特に「ルールの二重管理」「ファイル一覧」）

## 成果物の形と保存先
- Markdownの点検レポート1枚。保存先＝`output/日付_sync-check.md`
- 構成：①今回変わったファイル一覧 ②ペアごとの判定表（OK/要確認）③要確認の詳細 ④次にやること

## 合格条件（数えられる形）
1. ペア表の**全5ペア**について判定（OK / 要確認）が漏れなく出ている。
2. 「要確認」には**該当ファイル名が具体的に**書かれている（0件なら「ズレなし」と明記）。
3. 成果物が `output/日付_sync-check.md` として**1ファイル保存**されている。

## 正解例（要確認が1件あった場合の抜粋）
```
## ペアごとの判定
| ペア | 判定 |
|---|---|
| 1. cover-maker.html ⇄ docs/cover-maker-rules.md | ✅ OK（両方更新） |
| 3. kindle-cover-design-rules.md ⇄ 参照側3ファイル | ⚠️ 要確認 |
...
## 要確認の詳細
- ⚠️ docs/kindle-cover-design-rules.md は変わったが Agent.md が未更新。
  → 思想の原本を変えたら Agent.md も追従が必要（ペア#3）。
## 次にやること
- Agent.md を開き、原本の変更点が反映されているか確認する（人間の判断）。
```

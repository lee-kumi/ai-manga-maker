## 0. このプロジェクトの根っこ（全ツール共通の思想）

- これは「作品を完成させる場所」ではなく、**画像生成AI（ナノバナナ・ChatGPT）やテキストAI（Claude・ChatGPT）に渡すプロンプトを設計する場所**である
- 目的は**Kindle出版のためのAI漫画・表紙づくりを支援すること**。対象ユーザーは初心者
- 一番やりがちな失敗は、**中身（企画・キャラ）を固めずに、いきなり画像生成から始めること**。だからどのツールも「考える → 固める → 最後に画像生成」の順番を強制する
- **生成画像に日本語の文字は入れない**。文字は後から Canva で載せるのが全ツール共通の前提（日本語崩れ防止）
- **顔ブレ防止が品質の生命線**。キャラクターシートを作り、画像生成のたびに参照画像として添付する

## 2. ストーリー設計の知見（`story-design.html`）

企画の骨組みを、質問に答えていくだけで固めるツール。**入力データは外部に送信されず、ブラウザ内で完結する。**

### 設計の流れ（質問の順番）

1. **テーマ**：この漫画で「誰に・何を伝えるか」を1行で
2. **読者像（ペルソナ）**：年齢・状況・悩みを具体的に。届けたいのは**たった一人**
3. **読者の悩み**：3〜5つ書くと効果的
4. **ビフォー → アフター**：読む前と読んだ後の変化（例：「AIで漫画なんて無理」→「4コマ漫画を1本完成できた」）
5. **約束（ベネフィット）**：読者に約束できることを1行で言い切る
6. **構成の型を選ぶ**（下表）
7. **章立て**：章数を選び、各章のタイトル・役割・読後変化を決める

### 構成の型（4種類）

| 型 | 向いている用途 | ひとこと | 章の展開 |
|---|---|---|---|
| **起承転結** | SNS・4コマ・短編 | 一番シンプル。まずこれから | 起（悩み・現状）→ 承（AIと出会う・試す）→ 転（失敗・壁＝ここが命）→ 結（小さな成功・気づき） |
| **三幕構成** | Kindle・連載・長編 | 映画・ドラマと同じ構造 | 第1幕（日常・問題発生）→ 第2幕（葛藤・最大の試練）→ 第3幕（解決・新しい日常） |
| **問題解決型** | 集客・教育・商品紹介 | 売れる漫画はほぼこれ入ってる | 問題提起 → 原因分析 → 解決策（AI活用）→ 実証（結果が出る） |
| **ヒーローズジャーニー** | ファン化・感情重視 | 読者が「自分ごと」として読む | 日常世界 → 召喚（出会い）→ 試練（葛藤）→ 変容（成長）→ 帰還（新しい自分へ） |

> **共通の勘どころ**：どの型でも「転／試練＝失敗・壁」が物語の命。ここが弱いとただの日常になる。

### v2のSTEP1「企画書」で使う5要素（必須）

ストーリー設計をさらに絞り込むと、v2 の企画書は次の5要素になる。

1. **悩み**（出発点）— 主人公が最初にどんな状態で、何に悩んでいるか
2. **本音**（心の声）— 飾らない正直な気持ち。読者が共感するのは正直な気持ち
3. **壁**（挫折）— 途中でぶつかる問題。壁があることで物語に深みが出る
4. **カギ**（気づき）— 前に進むための気づき。何を学んで変わったのか
5. **結末** — 最後にどう変わるか＋読者に何を届けたいのか

（任意：事件・転機・成長。入れるほど具体的になる）

### タイトルの条件（優先順位順）

1. **【最優先】検索キーワードを含める** — 読者が実際に検索する言葉（「AI漫画」「50代から」「初心者でも」「〇日で」）。抽象的で気の利いたタイトルはNG
2. **【次点】ベネフィットを伝える** — 読むと何が得られるか（「〇日でできる」「素人でも作れる」）
3. **【補強】数字を入れて信頼感を上げる**（7日・3ステップ・90日）
4. 主人公の悩みや変化が伝わると尚良い

## 3. AI漫画メーカー v2 の知見（`v2.html`）— 要約

> 詳細は原本 [`docs/manga-maker-rules.md`](./manga-maker-rules.md) を参照。ここは要点だけ。

**基本思想：漫画は「絵」ではなく「物語」である。** 読者の心を動かすことが目的。必ず **企画 → キャラ・画風 → コマ割り → 画像生成** の順に進み、画像生成は最後。

### STEP別の要点

- **STEP1 企画**：上記「2. ストーリー設計の知見」の5要素・ペルソナ・タイトルがそのまま該当
- **STEP2 キャラ・画風**：画風は**全コマ共通で1つに固定**。キャラは**キャラクターシート**（→ 4章）で顔を固める
- **STEP3 コマ割りシナリオ**：
  - セリフは**句点（。）を使わない**・**1コマ30文字以内**・「セリフ／モノローグ／ナレーション」を分ける
  - スマホ読み前提で**1ページ1メッセージ**。1ページ3〜4コマが目安
  - クライマックスは**大ゴマ**で止める。最後のコマは**「めくり」**（次が読みたくなる所で切る）
  - レイアウトは毎ページ変え、同じパターンを**3ページ以上続けない**
- **STEP4 画像生成**（最重要2点）：
  1. コマの読む順序は必ず**「右上→左上→右下→左下」（日本式・右から左）**。左→右配置は絶対禁止
  2. **参照画像（キャラシート）の外見を全コマで忠実に再現**する
  - 文字は**縦書き**・吹き出し背景は**純白（#FFFFFF）**・ページ番号は画像内に描かない
  - 標準ネガティブプロンプト：`lowres, bad anatomy, bad hands, extra fingers, blurry, deformed, watermark, modern objects`

## 4. キャラクターシートの知見（`charsheet.html`）

**顔ブレ防止の要。** 各項目を入力すると、画像AI（ChatGPT・Gemini・ナノバナナ）に渡す**1枚もののキャラクターデザインシート作成プロンプト**が生成される。空欄はAIが自動補完する。

### 入力する設定項目

- **基本**：名前・年齢・性別・職業／立場
- **外見**：顔型・目の形・眉・鼻・口・肌の色
- **髪型**：髪型・髪色・前髪・長さ・特徴
- **服装**：上半身・下半身・靴・アクセサリー
- **性格**：性格・第一印象・特徴的な仕草／癖
- **画風**：漫画風・アニメ風・シンプルな線画 など（空欄でAI自動選択）

### 生成される1枚シートの構成（7点セット＋英語ラベル）

1. 全身正面（Front View）
2. 全身側面（Side View）
3. 全身背面（Back View）
4. 顔アップ（Face Close-up）
5. 表情集（Expressions）：通常・笑顔・怒り・悲しみ・驚き・困り顔
6. 小物設定（Props）：そのキャラがよく使う持ち物
7. カラーパレット（Color Palette）：髪色・肌色・服装・アクセントカラー

### レイアウトの指定

- **白背景**・キャラクターデザイン資料集形式・見やすく整理・**全身が切れない配置**・**高解像度**
- ラベルは英語表記（Character Design Sheet / Front View … Color Palette）

> このシートを、v2 STEP4 や表紙の画像生成のたびに**参照画像として必ず添付**する。これが全ツールを貫く顔ブレ防止の中心。

## 5. Kindle表紙メーカーの知見（`cover-maker.html`）— 要約

> 詳細は原本 [`docs/cover-maker-rules.md`](./cover-maker-rules.md) を参照。ここは要点だけ。

**基本思想：表紙はアートではなく「広告」である。** 目的はうまい絵ではなく、**一覧の中でクリックされ、内容が一瞬で伝わること**。必ず **本の情報（タイトル・ターゲット・フック）→ 表紙コンセプト → 画像生成** の順。

### 2つのプロファイル

| プロファイル | 対象 | 主役 | 重視する要素 |
|---|---|---|---|
| 🎨 AI漫画の表紙 | AI漫画作品 | イラスト・キャラ | 表情・ポーズ、画風、象徴的なシーン、感情 |
| 📖 Kindleテキスト本の表紙 | 文章主体の本 | タイトル・コピー | タイポグラフィ、配色、視認性、余白 |

### 表紙の鉄則（共通）

1. うまい絵より「クリックされること」を最優先
2. **スマホの小さなサムネイルで成立するか**が最重要（縮小してタイトルが読めなければ失敗）
3. タイトルに**句点（。）を付けない**
4. **色数は3色以内**（背景・文字・差し色）。絞るほど強く見える
5. 「誰の・どんな悩みが・どうなる本か」が1秒で伝わるか
6. 1秒で内容が伝わり、2秒目でクリックしたくなる情報設計
7. 同ジャンルの売れ筋と比べ、埋もれず・浮きすぎない位置を狙う

- 漫画表紙：キャラの表情・ポーズで感情を伝え、主役を一目で分かる大きさに。**キャラシートを参照画像に添付**して画風を統一
- テキスト本表紙：タイトルが主役（画面の30〜50%）。書体で印象が決まる（明朝＝知的、ゴシック太＝力強い、丸ゴシック＝やさしい）

## 6. 全ツール共通の技術ルール

| 項目 | ルール |
|---|---|
| 構成 | **1アプリ＝1つの自己完結HTMLファイル**。CSS/JSはすべてインライン。ビルド工程・外部ライブラリなし（Google Fontsのみ例外） |
| データ保存 | **localStorageのみ**。キーは `プレフィックス_作品名_項目` 形式（v2は `v2_`、表紙は `cm_`）。作品名に半角アンダースコア（_）は禁止 |
| バックアップ | どのアプリにもバックアップ保存／復元（JSONファイル書き出し）を付ける |
| キャンバスサイズ | 漫画本文＝**1000×1414px（A4縦）** / 表紙＝**1600×2560px（Kindle推奨・1.6:1）** |
| 文字の扱い | 生成画像に日本語文字は入れず、**後から Canva で載せる** |
| デザイン | Noto Sans JP、カード型UI、ポータルのアクセント色は `#E8612A` |
| UI文言 | 対象は初心者。日本語で、専門用語を避けて丁寧に |
